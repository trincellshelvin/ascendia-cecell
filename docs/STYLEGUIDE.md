# CeCell Style Guide  
### Code • Naming • UI/UX • Accessibility • Documentation

This guide ensures consistency, clarity, and accessibility across all CeCell contributions.

---

# 🧱 Code Style

### TypeScript Required
All source code must be written in TypeScript.

### Naming Conventions
- **camelCase** for variables and functions  
- **PascalCase** for classes and interfaces  
- **kebab-case** for file names  
- **UPPER_CASE** for constants  

### Function Guidelines
- Small, focused functions  
- Prefer async/await  
- Avoid deeply nested logic  
- Use descriptive names  

---

# 🎨 UI/UX Style

### Accessibility First
- High contrast  
- Keyboard navigable  
- Clear labels  
- Screen‑reader friendly  

### Layout
- Clean, minimal, structured  
- Avoid clutter  
- Prioritize clarity over decoration  

### Panels
- Consistent spacing  
- Clear section headers  
- Predictable interactions  

---

# 🎤 Voice Interaction Style

Commands should be:
- Short  
- Natural  
- Easy to pronounce  
- Error‑tolerant  

Examples:
- “Analyze workspace”  
- “Generate patch”  
- “Open memory panel”  

---

# 📝 Documentation Style

- Use plain, clear language  
- Prefer short paragraphs  
- Use lists for clarity  
- Include examples  
- Avoid jargon unless necessary  

---

# ❤️ Mission Alignment

Every line of code and every UI element should reflect:
- Accessibility  
- Autonomy  
- Clarity  
- Reliability  
- Legacy
